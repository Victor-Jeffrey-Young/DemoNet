## 阶段二：后端稳定性与性能优化测试用例

### 1. 消息队列与缓存稳定性
| 测试编号 | 测试场景 | 操作步骤 | 预期结果 |
| :--- | :--- | :--- | :--- |
| **BE-01** | **Redis 连接池防阻塞** | 1. 使用压测工具（如 JMeter 或 Apache ab）对任意使用缓存的接口并发发送 200 个请求 | Redis 连接稳定分配（Lettuce 激活多线程处理），接口无 `RedisConnectionFailureException`，平均响应时间波动较小。 |
| **BE-02** | **RabbitMQ 死信队列与重试机制** | 1. 故意向 RabbitMQ 队列发送格式错误或无法处理的消息体（例如损坏的抓取任务）<br>2. 观察控制台日志及 RabbitMQ 管理面板 | 消费失败一定次数后，消息不再陷入死循环无休止消耗 CPU，而是被路由转移至 `dlx` 死信交换机并停留在死信队列中等待手动排查。 |
| **BE-03** | **IGDB Token 缓存共享** | 1. 启动多个后端实例（或多次调用获取 IGDB Token 的方法） | 仅在第一次调用或 Token 即将过期时向 Twitch API 发起 HTTP 请求获取 Token，后续请求均直接从 Redis 获取，且多个实例共享同一个 Token。 |

### 2. N+1 瓶颈与 SQL 性能优化
| 测试编号 | 测试场景 | 操作步骤 | 预期结果 |
| :--- | :--- | :--- | :--- |
| **BE-04** | **AdminService 批量更新优化** | 1. 在管理后台勾选 100 条内容数据<br>2. 执行“批量上架/下架”操作<br>3. 监控后端打印的 SQL 语句 | 只打印 1 条执行语句：`UPDATE items SET status = ? WHERE id IN (...)`，而不再是 100 条单独的 `UPDATE` 和 `SELECT` 语句。操作耗时降至毫秒级。 |
| **BE-05** | **ReviewService 统计聚合优化** | 1. 对某个 ID 的条目（itemId=100）插入 5000 条测试评论数据<br>2. 调用 `/api/public/items/100` 或直接请求 `ReviewService.stats` | 只产生 1 条 SQL 语句 `SELECT COUNT(*), AVG(rating)...`，内存占用平稳，不触发大规模 GC，耗时在 10ms 以内（相比之前取出 5000 个 Java 对象的数百毫秒耗时大幅缩短）。 |

### 3. 第三方接口并发抓取优化
| 测试编号 | 测试场景 | 操作步骤 | 预期结果 |
| :--- | :--- | :--- | :--- |
| **BE-06** | **Steam 批量同步并发抓取** | 1. 触发 `SteamService.updateExistingGames()` 批量同步，或调用包含多个 DLC 的 Steam 游戏详情抓取 | 多个网络请求在 `CompletableFuture` 自定义线程池（大小 10）下并行执行，控制台同时打印多条处理日志，整体响应时间接近最慢单个请求的时间，而不是所有请求时间的累加。 |
| **BE-07** | **Steam 海报多重检测与缓存** | 1. 请求某个 Steam 游戏的海报图片（调用 `resolvePosterUrl`）<br>2. 多次请求相同的游戏 | 首次需要多次 HEAD 请求探测可用 CDN，解析出最终海报后，结果将写入 Redis 缓存。后续在 7 天内重复抓取时，瞬间返回缓存的海报链接。 |
| **BE-08** | **TMDB 搜索结果并发详情抓取** | 1. 在后台新增内容中搜索 TMDB 影视关键字（如“Matrix”）<br>2. 抓取列表渲染耗时 | 搜索获取到 10-20 条结果后，由于并发获取每一条记录的详情（`fetchMovieDetail` / `fetchTVDetail`），搜索列表加载速度相比之前的串行循环抓取提升 80% 以上。 |

## 阶段三：核心瓶颈突破与资源优化测试用例

### 1. 内存及 OOM 隐患消除
| 测试编号 | 测试场景 | 操作步骤 | 预期结果 |
| :--- | :--- | :--- | :--- |
| **BE-09** | **大字段排除测试 (Items)** | 1. 请求 `/api/items/hot` 或 `/api/items/featured` 等列表接口<br>2. 检查响应的 JSON 负载 | 返回的 Item 列表中仅包含 `id`, `title`, `cover_url` 等必要字段，不再包含臃肿的 `description` 和 `info_json` 字段。单次请求的数据体积从几 MB 降至几十 KB，彻底消除 OOM 风险。 |

### 2. 数据库索引与安全加固
| 测试编号 | 测试场景 | 操作步骤 | 预期结果 |
| :--- | :--- | :--- | :--- |
| **BE-10** | **关键索引验证** | 1. 检查数据库结构（如执行 `SHOW INDEX FROM reviews;`） | 成功应用 `V2__Add_indexes.sql` 脚本，表内正确生成了 `idx_reviews_item`, `idx_reviews_user`, `idx_items_status` 等关键索引，有效避免十万级数据下的全表扫描。 |
| **BE-11** | **防 SQL 注入验证** | 1. 检查 `ItemService` 与 `TagService` 中的自定义 SQL | 已完全废除原有的字符串拼接（如 `replace("'", "''")`），改为安全的 `?` 占位符与预编译参数绑定（`PreparedStatement`），阻断 SQL 注入路径。 |
| **BE-12** | **AdminController JSON 处理** | 1. 在后台进入“阅读条目”详情，上传或更新一张封面图片<br>2. 检查数据库中该条目的 `info_json` | `reader_url` 属性能够被正确写入或覆盖，且未破坏该字段内的其他 JSON 结构（已弃用危险的正则篡改，改用成熟的 `ObjectMapper` 解析机制）。 |

### 3. 高可用防护机制
| 测试编号 | 测试场景 | 操作步骤 | 预期结果 |
| :--- | :--- | :--- | :--- |
| **BE-13** | **RestClient 全局超时阻断** | 1. 断开网络或人为制造第三方（Steam/TMDB）极高延迟<br>2. 触发一次第三方条目搜索 | 强制在 5 秒（连接建立）或 15 秒（读取等待）后主动抛出超时异常并结束请求，杜绝无期限等待导致 Tomcat 线程池耗尽（假死）。 |
| **BE-14** | **文件上传上限防御** | 1. 尝试上传一个超过 10MB 的测试文件 | 服务端在入口层即刻拦截并返回 `413 Payload Too Large`，验证 `max-file-size=10MB` 配置已生效，封堵大文件恶意挤占磁盘与内存的漏洞。 |

## 阶段四：系统细节优化与底层防线巩固测试用例

### 1. Redis 与序列化安全加固
| 测试编号 | 测试场景 | 操作步骤 | 预期结果 |
| :--- | :--- | :--- | :--- |
| **BE-15** | **Redis 反序列化白名单拦截** | 1. 尝试将伪造的带有恶意 `@class`（如 `java.lang.Runtime` 等非项目类）的 JSON 缓存数据写入 Redis<br>2. 触发对应的读取接口 | 系统抛出反序列化类型不允许的异常（由 `BasicPolymorphicTypeValidator` 拦截），成功防御潜在的 RCE（远程代码执行）攻击。 |
| **BE-16** | **自定义缓存 TTL 策略验证** | 1. 触发 `hotItems` 和 `visibleTypes` 的查询接口以生成缓存<br>2. 登录 Redis（或使用监控工具）查看 TTL | `hotItems` 的 TTL 显示不超过 5 分钟，而 `visibleTypes` 的 TTL 显示长达 1 天（86400秒），不再是全局僵化的 10 分钟。 |

### 2. 对象复用与 CPU 性能优化
| 测试编号 | 测试场景 | 操作步骤 | 预期结果 |
| :--- | :--- | :--- | :--- |
| **BE-17** | **JWT 验证开销优化** | 1. 对受保护的接口发起并发压力测试（模拟高频校验 Token） | `JwtAuthFilter` 现在针对每次请求仅解密、解析一次 JWT 签名提取 `Claims`，消除了原本 3 次重复解密的性能浪费，CPU 负载显著降低。 |
| **BE-18** | **ObjectMapper 实例复用** | 1. 执行包含第三方数据合并更新的操作（频繁触发 `ItemService.mergeInfoJson`） | 业务逻辑复用了 Spring 容器注入的 `ObjectMapper`，不再局域实例化重量级对象，避免了高并发下的短命对象激增及 Minor GC 卡顿。 |
| **BE-19** | **SteamGridDB 密钥内存缓存** | 1. 触发后端后台批量同步 Steam 游戏海报的操作 | `SteamGridDBService` 只会在首个请求时触发 1 次 `SELECT` 获取 API 密钥，在随后的 5 分钟内数以百计的调用均直接从内存返回，免除冗余的数据库 I/O。 |

### 3. 数据完整性与带宽优化
| 测试编号 | 测试场景 | 操作步骤 | 预期结果 |
| :--- | :--- | :--- | :--- |
| **BE-20** | **静态资源浏览器强缓存** | 1. 打开开发者工具 (Network)<br>2. 访问携带 `uploads` 目录下图片的页面<br>3. 刷新页面 | 首次请求返回 200，后续刷新图片请求显示 `(disk cache)` 或 `(memory cache)` 或返回 304。由于 `WebConfig` 配置了 `86400`，图片被正确缓存在本地，降低了服务器出口带宽压力。 |
| **BE-21** | **外键级联与脏数据清理** | 1. 执行后台删除某个 User（包含他留下的评论和收藏）<br>2. 查阅关联表 `reviews` / `user_items` | `V3__Add_foreign_keys.sql` 成功执行并清理了历史脏数据。得益于 `ON DELETE CASCADE`，主表数据删除后，从表的孤立评论/收藏记录被数据库原生机制自动同步销毁，保证数据绝对一致性。 |
