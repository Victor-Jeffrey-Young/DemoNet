# 建议与问题汇总

> **来源**：2026-05-28 项目代码审查
> **状态**：待处理

---

## 一、安全隐患（高优先级）

### 1.1 SQL 拼接注入风险

- **位置**：`ItemService.listRecommended()` 和 `getItemIdsByTags()`
- **现象**：使用字符串拼接构建 SQL，虽然有 `replace("'", "''")` 做简单转义，但这不是可靠的防护手段
- **建议**：改用 `JdbcTemplate` 的 `?` 占位符参数化查询，或 MyBatis-Plus 的 `LambdaQueryWrapper` 条件构造

### 1.2 Admin 路由无前端守卫

- **位置**：`router/index.js` 的 `/admin` 路由
- **现象**：未设置 `meta: { requiresAuth: true }`，匿名用户可直接访问管理面板页面
- **建议**：添加 `meta: { requiresAuth: true, requiresAdmin: true }` 并在 `beforeEach` 守卫中校验角色

### 1.3 Admin 接口无后端角色校验

- **位置**：`AdminController.java`（`/api/admin/**`）
- **现象**：`SecurityConfig` 对 `/api/admin/**` 只要求"已认证"，普通登录用户也能触发 Steam/TMDB 抓取和内容审核
- **建议**：`User` 实体增加 `role` 字段（`USER`/`ADMIN`），`SecurityConfig` 对 `/api/admin/**` 要求 `ADMIN` 角色

### 1.4 敏感信息硬编码

- **位置**：`docker-compose.yml`、`application.yml`
- **现象**：MySQL root 密码 `changeme`、RabbitMQ 密码 `changeme`、JWT secret 均硬编码在配置文件中
- **建议**：使用环境变量（`${MYSQL_ROOT_PASSWORD}`）或 `.env` 文件管理，`.env` 加入 `.gitignore`

## 二、前端 Bug

### 2.1 401 重定向失效

- **位置**：`api/request.js` 响应拦截器
- **现象**：使用 `window.location.hash = '#/login'` 进行跳转，但路由使用 `createWebHistory()`（history 模式），hash 变化不会触发路由导航
- **影响**：token 过期后用户不会被引导到登录页
- **建议**：改用 `router.push('/login')` 或 `window.location.href = '/login'`

### 2.2 动态 Tailwind 类名不生效

- **位置**：`List.vue` 分页按钮、`Search.vue` 分页按钮
- **现象**：`bg-${meta.accent}-600` 这种运行时拼接的类名不会被 Tailwind JIT 编译器扫描到，导致样式缺失
- **建议**：使用 `safelist` 配置预生成需要的颜色类，或在 `constants/types.js` 中直接存储完整类名（如 `bgColor: 'bg-emerald-600'`）

### 2.3 Element Plus 全量引入但未使用

- **位置**：`main.js` 中 `import ElementPlus from 'element-plus'`
- **现象**：所有 UI 均使用 Tailwind 构建，Element Plus 未被使用但增加了约 1MB+ 打包体积
- **建议**：移除 Element Plus 依赖，或改为按需引入（`unplugin-vue-components` + `unplugin-auto-import`）

## 三、后端问题

### 3.1 无全局异常处理

- **现象**：Service 层抛出 `RuntimeException` 会返回 HTTP 500 + Java 堆栈信息给前端
- **建议**：创建 `GlobalExceptionHandler`（`@RestControllerAdvice`），定义业务异常类（如 `BusinessException`），统一返回 `{ code, message }` 格式

### 3.2 无输入校验

- **现象**：所有 Controller 接口都没有参数校验（`@Valid`），注册接口可以接受空用户名、空密码、格式错误的邮箱
- **建议**：Entity 或 DTO 上添加 `@NotBlank`、`@Email`、`@Size` 等注解，Controller 参数加 `@Valid`

### 3.3 Entity 直接暴露给前端

- **现象**：`User` 实体包含 `passwordHash` 字段，JSON 序列化时会返回给前端（虽然 BCrypt 哈希不可逆，但仍暴露内部结构）
- **建议**：引入 DTO 层（`UserDTO`、`ItemDTO`），或在 Entity 字段上添加 `@JsonIgnore`

### 3.4 Entity/DB 列不同步

- **位置**：`Item.java` ↔ `items` 表
- **现象**：2026-05-28 新增 `posterUrl` 字段后整站瘫痪，MyBatis-Plus 生成的 SQL 引用不存在的列 `poster_url`（详见 [问题日志 #29](./11-问题日志.md#29-poster_url-列缺失导致整站瘫痪)）
- **根因**：`schema.sql` 不会对已存在的表自动执行 ALTER；Entity 变更后缺数据库迁移机制
- **建议**：引入 Flyway 或 Liquibase 管理 schema 变更；或在每次 Entity 变更后编写对应的 ALTER 脚本

### 3.5 推荐查询使用原生 SQL

- **位置**：`ItemService.listRecommended()`
- **现象**：手动拼接 SQL + 手动映射 12 个字段，与项目其他地方的 MyBatis-Plus 风格不一致，维护成本高
- **建议**：改用 MyBatis-Plus 的 `selectPage` + 自定义 SQL，或将排序逻辑抽象为 Service 方法

## 四、前端架构问题

### 4.1 类型元数据重复定义

- **现象**：品类信息（emoji、标签、颜色、描述）在以下四处各自维护，格式不一致：
  1. `constants/types.js` — `TYPE_META`（主定义）
  2. `AppCard.vue` — 本地 `meta` 对象
  3. `Detail.vue` — `typeMetaMap`
  4. `Home.vue` — `categories` 数组
- **建议**：所有组件统一从 `constants/types.js` 导入 `TYPE_META` / `getMeta()`，删除其余三处重复定义

### 4.2 缺少数据层缓存

- **现象**：每个页面独立请求数据，`Home.vue` 和 `List.vue` 都各自请求热门列表，无缓存复用
- **建议**：创建 `stores/item.js`（Pinia），统一管理 items 数据，支持请求缓存和跨组件共享

### 4.3 加载状态和错误处理不完善

- **现象**：
  - 各页面用纯文字"加载中..."作为加载状态，无骨架屏
  - 多处空 catch 块（`catch {}` 或 `catch (e) { /* */ }`）吞掉错误，难以调试
- **建议**：添加骨架屏组件替代纯文字加载态；创建全局错误处理（如 Toast 提示），避免静默失败

### 4.4 SearchBar 回车不绕过防抖

- **位置**：`SearchBar.vue`
- **现象**：按 Enter 键仍然等待 300ms 防抖延迟，没有立即触发搜索
- **建议**：`@keydown.enter` 事件处理中直接执行搜索，跳过 debounce 定时器

## 七、总体评价

**方向清晰、文档扎实、架构合理**。品类差异化的视觉策略是亮点，在同类聚合平台中具有辨识度。

**主要短板**：
- 安全细节和防御性编程（SQL 拼接、无输入校验、无全局异常处理、无角色权限）—— 面向公网前必须解决
- 代码规范化（DTO 层、类型元数据统一、测试覆盖）—— 决定品类扩展时代码是否会越写越乱

**优先修复顺序建议**：安全隐患 > 前端 Bug > 后端架构问题 > 前端架构问题 > 中期基建 > 长期优化
