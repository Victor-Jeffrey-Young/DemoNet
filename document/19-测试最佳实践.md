# 后端测试最佳实践

## 概述

本文档描述 DemoNet 后端测试的标准模式,确保测试代码的可维护性、一致性和可靠性。

---

## 1. 认证 Mock 标准模式

### ❌ 不推荐:手动创建 Authentication Mock

```java
// 反例:每个测试类重复 mockAuth() 方法
private Authentication mockAuth(Long userId) {
    Authentication auth = mock(Authentication.class, withSettings().lenient());
    when(auth.getPrincipal()).thenReturn(userId);
    return auth;
}

// 在每个测试方法中调用
mockMvc.perform(post("/api/reviews")
    .with(authentication(mockAuth(1L)))
    .with(csrf()))
```

**问题**:
- 重复代码 (每个需要认证的 Controller 测试都要写一遍)
- 使用了不必要的 `AuthenticationPrincipalArgumentResolver`
- 每个测试方法调用都需要创建新的 mock 对象

### ✅ 推荐:使用 TestAuthentication 工具类

```java
import static com.example.demonet.test.TestAuthentication.*;

// 在 @BeforeEach 中初始化 (类级别复用)
private Authentication auth;

@BeforeEach
void setUp() {
    auth = withUserId(1L);
    mockMvc = MockMvcBuilders.standaloneSetup(new ReviewController(reviewService))
            .setControllerAdvice(new GlobalExceptionHandler())
            .build();
}

// 在测试方法中直接注入
mockMvc.perform(post("/api/reviews")
    .principal(auth)
    .contentType(MediaType.APPLICATION_JSON)
    .content("{\"itemId\":100,\"rating\":5}"))
```

**优势**:
- 消除重复代码 (工具类统一管理)
- 使用标准 MockMvc `.principal()` 方法,无需自定义 ArgumentResolver
- 类级别复用 mock 对象,减少对象创建开销

## 2. 带角色的认证 Mock

### 场景:测试管理员端点

```java
@Test
void admin_deleteItem_asAdmin_shouldSucceed() throws Exception {
    Authentication adminAuth = withAdmin(1L);

    mockMvc.perform(delete("/api/admin/items/100")
        .principal(adminAuth))
        .andExpect(status().isOk());
}
```

### 场景:测试普通用户权限

```java
@Test
void admin_deleteItem_asUser_shouldFail() throws Exception {
    Authentication userAuth = withUserIdAndRole(2L, "ROLE_USER");

    mockMvc.perform(delete("/api/admin/items/100")
        .principal(userAuth))
        .andExpect(status().isForbidden());
}
```

## 3. Standalone MockMvc vs. Full MockMvc

### Standalone MockMvc (推荐用于单元测试)

```java
mockMvc = MockMvcBuilders.standaloneSetup(new ReviewController(reviewService))
        .setControllerAdvice(new GlobalExceptionHandler())
        .build();
```

**特点**:
- 轻量级,启动快 (毫秒级)
- 只加载被测试的 Controller
- 需要手动注入依赖 (如 `GlobalExceptionHandler`)
- **认证注入使用 `.principal(auth)`**

### Full MockMvc (推荐用于集成测试)

```java
mockMvc = MockMvcBuilders.webAppContextSetup(context).build();
```

**特点**:
- 加载完整 Spring 上下文 (秒级)
- 自动应用所有配置 (Security, ArgumentResolvers 等)
- 适合测试过滤器链、Security 配置等
- 认证注入可以使用 `@WithMockUser` 注解

## 4. Mockito 严格模式与宽松模式

### 原则:默认使用严格模式,仅在必要时使用宽松模式

```java
// ✅ 严格模式 (默认)
when(restClient.post()).thenReturn(requestBodyUriSpec);
when(requestBodyUriSpec.uri(anyString())).thenReturn(requestBodyUriSpec);
// ... 所有 stub 都会被验证

// ⚠️ 宽松模式 (仅在非必经路径使用)
@Test
void emptyResponse_scenario() {
    mockEmptyResponse(); // 内部使用 lenient()

    Item result = service.fetchItem(100);
    assertThat(result).isNull();
}
```

### 何时使用 `lenient()`

1. **空响应测试**: 当测试场景根本不触发某个调用链时
2. **继承的基类方法**: 子类不需要的父类 mock 配置
3. **条件分支的未覆盖路径**: 如 `@BeforeEach` 中的通用配置

### ❌ 错误示例:过度使用 lenient()

```java
@Test
void badExample() {
    // 反例:全部使用 lenient(),掩盖了测试设计问题
    lenient().when(a).thenReturn(b);
    lenient().when(c).thenReturn(d);
    // ...
}
```

## 5. RestClient Mock 链优化

### 问题:RestClient 链式调用导致大量重复 stub

```java
// ❌ 冗长的原始 stub
when(restClient.post()).thenReturn(requestBodyUriSpec);
when(requestBodyUriSpec.uri(anyString())).thenReturn(requestBodyUriSpec);
when(requestBodyUriSpec.body(any(Object.class))).thenReturn(requestBodyUriSpec);
when(requestBodyUriSpec.header(anyString(), any())).thenReturn(requestBodyUriSpec);
when(requestBodyUriSpec.retrieve()).thenReturn(responseSpec);
when(responseSpec.body(String.class)).thenReturn(responseBody);
```

### ✅ 解决方案:使用 BaseIGDBServiceTest 辅助方法

```java
// 简单场景:缓存命中
mockTokenCache("cached-token");
mockRestClientChain("{\"id\":100,\"name\":\"Test\"}");

// 空响应场景
mockTokenCache("cached-token");
mockEmptyResponse();
```

### 复杂场景:完整 OAuth 流程

```java
@Test
void complexOAuthFlow() {
    // 1. token 缓存未命中
    mockTokenCache(null);

    // 2. Mock DB credentials
    when(jdbcTemplate.queryForObject(...)).thenReturn("client-id");

    // 3. Mock OAuth token 请求
    when(restClient.post()).thenReturn(requestBodyUriSpec);
    when(requestBodyUriSpec.uri(eq(TOKEN_URL))).thenReturn(requestBodyUriSpec);
    when(requestBodyUriSpec.body(any())).thenReturn(requestBodyUriSpec);
    when(requestBodyUriSpec.retrieve()).thenReturn(responseSpec);
    when(responseSpec.body(any(ParameterizedTypeReference.class)))
        .thenReturn(Map.of("access_token", "token", "expires_in", 3600));

    // 4. Mock 业务请求
    when(responseSpec.body(eq(String.class))).thenReturn(gameResponse);

    // 执行测试
    List<Item> results = service.searchGames("Test");
}
```

## 6. 测试基类设计原则

### ✅ 推荐:提供有意义的辅助方法

```java
abstract class BaseServiceTest {
    // 明确描述行为的辅助方法
    protected void mockTokenCache(String token) { ... }
    protected void mockRestClientChain(String response) { ... }
    protected void mockEmptyResponse() { ... }
}
```

### ❌ 避免:暴露原始 Mock 对象

```java
// 反例:让子类直接操作 Mock 对象
abstract class BadBaseTest {
    protected MockObject mock; // 子类需要了解内部实现
}
```

## 7. 断言最佳实践

### 使用 AssertJ 流畅断言

```java
// ✅ 流畅断言
assertThat(results)
    .hasSize(1)
    .first()
    .extracting(Item::getTitle)
    .isEqualTo("Test Game");

// ❌ 传统断言 (可读性差)
assertEquals(1, results.size());
assertEquals("Test Game", results.get(0).getTitle());
```

### 验证 Mock 交互

```java
// 验证方法是否被调用
verify(service).create(1L, 100L, 5, "Great!");

// 验证方法未被调用 (重要!)
verify(requestBodyUriSpec, never()).body(any()); // 搜索模式不发送 body

// 验证调用次数
verify(service, times(2)).getItem(anyLong());
verify(service, never()).delete(anyLong());
```

## 8. 常见反模式与解决方案

| 反模式 | 问题 | 解决方案 |
|--------|------|---------|
| 每个测试方法重复 mock 相同依赖 | 维护成本高,容易不一致 | 提取到 `@BeforeEach` 或基类辅助方法 |
| 使用 `anyString()` 匹配所有参数 | 测试不精确,容易掩盖错误 | 使用 `eq()` 或特定匹配器 |
| 忽略 `UnnecessaryStubbingException` | 测试代码冗余,可能遗漏重要 stub | 重构测试,移除不必要的 stub |
| 在 `@BeforeEach` 中全局 `lenient()` | 掩盖真实的测试问题 | 仅在特定测试方法中使用 `lenient()` |
| 硬编码 mock 返回值 (如 `1L`) | 难以测试不同场景 | 使用参数化测试或测试数据构建器 |

## 9. 测试命名规范

```java
// ✅ 清晰描述测试场景和预期结果
@Test
void createReview_withValidData_returnsReview() { ... }

@Test
void createReview_withInvalidRating_returns400() { ... }

@Test
void deleteReview_asOwner_succeeds() { ... }

@Test
void deleteReview_asDifferentUser_returns403() { ... }

// ❌ 模糊的测试名称
@Test
void testCreate() { ... }
@Test
void testDelete() { ... }
```

**命名模式**: `<方法名>_<场景>_<预期结果>`

## 10. Controller 测试检查清单

在提交 Controller 测试前,确认以下事项:

- [ ] 使用 `TestAuthentication.withUserId()` 创建认证 mock
- [ ] 使用 `.principal(auth)` 注入认证,而非 `.with(authentication())`
- [ ] 移除了 `.with(csrf())` (单元测试不需要 CSRF)
- [ ] 移除了 `AuthenticationPrincipalArgumentResolver` (Standalone MockMvc 不需要)
- [ ] 测试了参数校验失败场景 (返回 400)
- [ ] 测试了权限不足场景 (返回 403,如需要)
- [ ] 测试了认证未提供场景 (返回 401,如需要)
- [ ] 使用 `MockMvcBuilders.standaloneSetup()` 而非 `webAppContextSetup()`

---

## 11. Service 测试检查清单

在提交 Service 测试前,确认以下事项:

- [ ] 继承 `BaseIGDBServiceTest` (如测试 IGDBService)
- [ ] 使用 `mockTokenCache()` 而非重复 stub Redis
- [ ] 使用 `mockRestClientChain()` 或 `mockEmptyResponse()`
- [ ] 仅在必要时使用 `lenient()`,并添加注释说明原因
- [ ] 验证 Mock 交互 (使用 `verify()`)
- [ ] 无 `UnnecessaryStubbingException`
