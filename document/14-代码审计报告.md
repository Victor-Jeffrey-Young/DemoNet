## 二、安全问题（Security Issues）

### SEC-1 JWT 密钥硬编码在 `application.yml` 中

**文件位置**：`backend/src/main/resources/application.yml` 第 42 行

**问题描述**：真实的 JWT 密钥被提交到了 Git 仓库。

```yaml
42:42:backend/src/main/resources/application.yml
    secret: ${JWT_SECRET:T6KWuFG5rTmj5H1VD8sTqDrxn520O4Xqc83KYR5lTNP+IKt+BaeXpaDcp+3kbzslBdB5EjZ/cqXOxkFS4QvnvA==}
```

**影响分析**：任何能访问代码仓库的人都可以伪造管理员 JWT 令牌，完全绕过认证。即使后续通过环境变量覆盖，旧密钥仍留在 Git 历史中。

**修复建议**：
1. **立即轮换密钥**，生成新的高强度密钥
2. 从配置文件中移除默认值，仅通过环境变量注入：`secret: ${JWT_SECRET}`
3. 使用 `git filter-repo` 或 BFG 清理 Git 历史中的密钥

**修复建议**：
- 生产环境移除 MySQL/Redis/RabbitMQ 的端口映射，仅通过 Docker 内部网络通信
- 如需管理访问，使用 SSH 隧道而非直接端口映射

### SEC-4 上传文件无扩展名白名单验证

**文件位置**：`backend/src/main/java/com/example/demonet/controller/AdminController.java` 第 188-195 行

**问题描述**：虽然验证了 `Content-Type`，但文件扩展名完全来自用户输入。攻击者可以上传一个 `Content-Type: image/jpeg` 但实际是 `.html` 或 `.svg`（含 XSS 脚本）的文件。

```java
188:195:backend/src/main/java/com/example/demonet/controller/AdminController.java
            String originalName = file.getOriginalFilename();
            String ext = "";
            if (originalName != null && originalName.contains(".")) {
                ext = originalName.substring(originalName.lastIndexOf("."));
            }
            String filename = UUID.randomUUID().toString() + ext;
            Path filePath = uploadPath.resolve(filename);
            file.transferTo(filePath.toFile());
```

**影响分析**：可能导致存储型 XSS 攻击。若上传的 `.html`/`.svg` 文件可通过 URL 直接访问，攻击者可注入恶意脚本窃取用户 Cookie。

**修复建议**：
- 对文件扩展名做白名单校验（如仅允许 `.jpg/.jpeg/.png/.gif/.webp`）
- 校验文件魔数（magic bytes）而非仅依赖 Content-Type
- 配置 Nginx 对上传目录禁用脚本执行

### ARCH-3 `DemoNetApplication.clearCaches()` 使用 `redisTemplate.keys()`

**文件位置**：`backend/src/main/java/com/example/demonet/DemoNetApplication.java` 第 23-31 行

**问题描述**：`KEYS` 命令会阻塞 Redis 主线程，在 key 数量大时会导致服务不可用。

```java
23:31:backend/src/main/java/com/example/demonet/DemoNetApplication.java
    @PostConstruct
    public void clearCaches() {
        Set<String> keys = redisTemplate.keys("hotItems*");
        if (keys != null && !keys.isEmpty()) redisTemplate.delete(keys);
        keys = redisTemplate.keys("featured*");
        // ...
    }
```

**影响分析**：Redis 是单线程模型，`KEYS` 命令在 key 数量多时会阻塞数秒，期间所有缓存请求超时，可能导致后端雪崩。

**修复建议**：改用 `SCAN` 命令迭代删除：

```java
Set<String> keys = new HashSet<>();
try (Cursor<byte[]> cursor = redisTemplate.getConnectionFactory()
        .getConnection().scan(ScanOptions.scanOptions().match("hotItems*").count(100).build())) {
    while (cursor.hasNext()) {
        keys.add(new String(cursor.next()));
    }
}
if (!keys.isEmpty()) redisTemplate.delete(keys);
```

**文件位置**：`backend/src/main/java/com/example/demonet/service/AuthService.java` 第 37-67 行

**问题描述**：所有业务错误都抛 `RuntimeException`，然后被 `GlobalExceptionHandler` 统一捕获为 400 Bad Request。

### QUAL-2 `GlobalExceptionHandler` 将所有 `RuntimeException` 返回 400

**文件位置**：`backend/src/main/java/com/example/demonet/controller/GlobalExceptionHandler.java` 第 43-48 行

**问题描述**：数据库连接失败、空指针等服务器内部错误也会被包装成 400 返回给客户端。

```java
43:48:backend/src/main/java/com/example/demonet/controller/GlobalExceptionHandler.java

                .body(ApiResponse.error(400, e.getMessage() != null ? e.getMessage() : "请求处理失败"));
```

**影响分析**：
- 掩盖了真正的 500 错误，监控系统无法正确告警

- 安全隐患：可能泄露内部错误信息（如 SQL 异常细节）

**修复建议**：
- 仅捕获自定义 `BusinessException` 返回 400
- 其他 `RuntimeException` 返回 500 并记录完整堆栈
- 生产环境不向客户端返回原始异常消息
