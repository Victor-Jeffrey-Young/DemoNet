# 后端架构分析历史详表

> 本附录保留 2026-07-04 的 23 项后端风险分析全文，用于追溯当时的判断依据与改造背景。各项的当前状态以本报告“历史问题合并状态”和“当前问题清单”为准。

# 11 - 后端架构分析

> 分析日期：2026-07-04
> 目标：评估现有后端代码在小型服务器 + 数据量增长场景下的性能与安全风险

---

## 总体评估

代码功能完整，但**数据量涨到几千条 items + 上万条 reviews 时，小服务器大概率出问题**。核心瓶颈集中在三个方向：

1. 无分页的全量查询 — 内存 / 带宽压力
2. 外部 API 调用无超时 — Tomcat 线程池耗尽风险
3. 数据库缺关键索引 — 全表扫描

---

## 架构概览

```
Spring Boot 4.0.6 + MyBatis-Plus 3.5.16 + MySQL 8.0 + Redis 7 + RabbitMQ 3.12
```

- Service 层直接调用 8 个外部 API（Steam / TMDB / IGDB / AniList / Bangumi / iTunes / SteamGridDB / Cloudflare Turnstile），均通过 `RestClient` 且无超时
- 缓存仅覆盖 `visibleTypes` / `hotItems` / `featured` / `recommended` 四个键，统一 10min TTL
- RabbitMQ 用于异步抓取外部数据，7 个队列，无 DLQ / 重试 / 限流
- AdminController 25KB，直接注入 Mapper 绕过 Service 层，职责边界模糊

---

## 高危问题（必须修 — 会挂）

### 1. 列表接口无分页，全量返回

| 端点 | 位置 | 影响 |
|------|------|------|
| `GET /api/items/types?type=xxx` | `ItemService.listByType()` | 某类型下数百条 Item 全量返回，含 TEXT + JSON 大字段 |
| `GET /api/user/items` | `UserItemService.listByUser()` | 用户收藏无 LIMIT，重度用户数百条 |
| `GET /api/admin/users` | `UserMapper.selectUserList()` | 全量返回用户列表，无 LIMIT |
| `GET /api/admin/invite-codes` | `InviteCodeMapper.selectWithUsers()` | LEFT JOIN 全表返回 |
| `GET /api/tags` | `TagService.listAll()` | 全量返回标签 |

**数据量过千时**：单次响应几 MB，JVM 堆内存飙升，Tomcat 线程阻塞。

**修复**：所有列表端点改为分页查询（MyBatis-Plus `Page<>`），接收 `page` / `size` 参数。

---

### 2. Item 列表返回 description(TEXT) + infoJson(JSON) 大字段

**位置**：所有返回 `List<Item>` 的端点（hot / featured / recommended / byType / admin items）

**影响**：`infoJson` 含 screenshots 数组，单条可达数十 KB。列表 100 条 = 几 MB 的冗余 JSON。`description` 为 TEXT 字段，列表场景完全不需要。

**修复**：创建 `ItemSummaryDTO`（仅含 id / title / slug / coverUrl / type 等摘要字段），列表端点使用 DTO 投影，或在 MyBatis-Plus 查询中 `select()` 排除大字段。

---

### 3. ReviewService.stats() 全量加载评论算平均分

**位置**：`ReviewService.java:40`

```java
// 当前：加载所有 Review 对象到内存再算 AVG
List<Review> reviews = reviewMapper.selectList(qw);
double avg = reviews.stream().mapToInt(Review::getRating).average().orElse(0);
```

**影响**：一个热门 item 5000 条评论 → 全部 SELECT 到 JVM 堆，只为算一个平均值。

**修复**：改为 SQL 聚合：
```sql
SELECT COUNT(*) as count, AVG(rating) as avg_rating FROM reviews WHERE item_id = ?
```

---

### 4. SQL 注入漏洞（2处）

#### 4a. TagService.getItemIdsByTagNames()

**位置**：`TagService.java:90`

```java
// 标签名直接拼入 SQL，零转义
tagNames.stream().map(s -> "'" + s + "'").toList()
```

攻击者构造恶意标签名可注入任意 SQL。

#### 4b. ItemService.listHotItems()

**位置**：`ItemService.java:123`

```java
// 手动 replace 单引号，无法防御所有注入场景
sql += " AND i.type = '" + type.replace("'", "''") + "'";
```

**修复**：全部改用 `?` 占位符 + `PreparedStatement` 参数化绑定。

---

### 5. 8 个外部 API 全部无超时配置

**位置**：所有外部 API Service 中的 `RestClient.create()`

| Service | 外部 API |
|---------|----------|
| `SteamService` | Steam Store API |
| `TMDBService` | TMDB API |
| `IGDBService` | IGDB API (Twitch) |
| `AniListService` | AniList GraphQL |
| `BangumiService` | Bangumi API |
| `ItunesService` | iTunes Search API |
| `SteamGridDBService` | SteamGridDB API |
| `AuthService` | Cloudflare Turnstile |

**影响**：外部 API 慢响应或 502 时，Tomcat 线程被无限阻塞。默认线程池 200 个，几十个并发慢请求即可耗尽，导致**整个服务不可用**。

**修复**：创建统一的 `RestClient` Bean，配置 `connectTimeout: 5s`、`readTimeout: 15s`，所有 Service 注入此 Bean。

---

### 6. 数据库缺关键索引

**位置**：`V1__Baseline.sql`

当前 Schema 中以下高频查询字段**无索引**：

| 表 | 缺失索引 | 影响 |
|----|----------|------|
| `reviews` | 无 `item_id` 索引 | 按 item 查评论 → 全表扫描 |
| `reviews` | 无 `user_id` 索引 | 按用户查评论 → 全表扫描 |
| `user_items` | 无 `(user_id, item_id)` 联合唯一索引 | 收藏查询全表扫描 + 可能产生重复数据 |
| `items` | 无 `status` 索引 | 所有公开查询 `WHERE status=1` 全表扫描 |
| `item_tag_mapping` | 无 `tag_id` 反向索引 | 删标签时全表扫描（联合主键只优化 `item_id` 前缀） |

**修复**：新增迁移脚本 `V2__Add_indexes.sql`：

```sql
CREATE INDEX idx_reviews_item ON reviews (item_id);
CREATE INDEX idx_reviews_user ON reviews (user_id);
CREATE UNIQUE INDEX idx_user_item ON user_items (user_id, item_id);
CREATE INDEX idx_items_status ON items (status);
CREATE INDEX idx_itm_tag ON item_tag_mapping (tag_id);
```

## 中危问题（应该修 — 会慢）

### 7. N+1 循环查询：batchUpdateStatus

**位置**：`AdminService.java:90-99`

```java
for (Long id : ids) {
    Item item = itemMapper.selectById(id);  // 第 1 次查询
    item.setStatus(status);
    itemMapper.updateById(item);             // 第 2 次查询
}
```

批量更新 100 条 = 200 次 DB 往返。

**修复**：改为单条批量 SQL：
```java
new LambdaUpdateWrapper<Item>().in(Item::getId, ids).set(Item::getStatus, status);
itemMapper.update(null, wrapper);
```

---

### 8. Steam 导入串行 HTTP（N+1 外部调用）

**位置**：`SteamService.fetchByAppIds()` + `resolvePosterUrl()`

- `fetchByAppIds()` 循环内逐个调用 `fetchAppDetail()`
- `resolvePosterUrl()` 每个游戏最多 6 次 HTTP（4 次 HEAD + SteamGridDB）
- `extractDLC()` 每个 DLC 单独 1 次 HTTP（最多 5 个）

**影响**：导入 50 个游戏 = 300+ 次串行 HTTP 请求。

**修复**：使用 `CompletableFuture` 并行化（控制并发度 3-5），海报 URL 缓存到 Redis。

---

### 9. TMDB 搜索 N+1 外部调用

**位置**：`TMDBService.searchMovies()`

搜索后循环内逐个 `fetchMovieDetail()`。TMDB 搜索默认返回 20 条 = 20 次串行 HTTP。

**修复**：并行化请求，利用搜索结果中已有字段减少 detail 调用。

---

### 10. RabbitMQ 无容错机制

**位置**：`FetchConsumer` + `RabbitMQConfig`

- 默认 AUTO ack，消费失败无限 requeue（无限重试）
- 无 DLQ（死信队列）
- 无 prefetch 限制，消息堆积时可能 OOM

**修复**：
- 队列配置 DLX（死信交换机）+ DLQ
- 设置 `spring.rabbitmq.listener.simple.prefetch=5`
- 设置 `retry.enabled=true` + `max-attempts=3`

---

### 11. listHotItems 相关子查询

**位置**：`ItemService.java` SQL

```sql
(SELECT COUNT(*) FROM reviews r WHERE r.item_id = i.id
 AND r.created_at > DATE_SUB(NOW(), INTERVAL 7 DAY))
```

对每行 item 做一次 reviews 子查询 → O(N×M) 扫描。

**修复**：改为 LEFT JOIN + GROUP BY，或在 items 表维护 `recent_review_count` 字段定期刷新。加 `reviews.item_id` 索引后子查询性能也会大幅改善。

---

### 12. Redis 无连接池

**位置**：`application.yml`

仅配置 `host` / `port`，Lettuce 默认单连接，高并发下排队。

**修复**：
```yaml
spring:
  redis:
    lettuce:
      pool:
        max-active: 16
        max-idle: 8
        min-idle: 2
```

---

### 13. 文件上传 100MB 无额外限制

**位置**：`application.yml:11-12`

`max-file-size` 和 `max-request-size` 均 100MB，恶意用户可上传大文件耗尽磁盘 / 内存。

**修复**：降低上限至 10MB（图片足够），Controller 层增加文件类型校验。

---

### 14. IGDB token 存实例字段

**位置**：`IGDBService.java:32-34`

`accessToken` 和 `tokenExpiry` 存在实例字段中，多节点部署时每个节点独立获取 token。`synchronized` 在高并发时造成线程争用。

**修复**：将 token 缓存到 Redis（`key=igdb:token`，`TTL=expiry-60s`）。

---

### 15. SteamGridDBService 每次调用查数据库取 API key

**位置**：`SteamGridDBService.java:29-36`

`getApiKey()` 每次调用执行 `SELECT setting_value FROM app_settings`。在 `backfillPosterUrls` 循环中被频繁调用。

**修复**：使用 `@Cacheable` 注解缓存 API key（TTL 5 分钟）。

---

### 16. GenericJackson2JsonRedisSerializer 反序列化风险

**位置**：`RedisConfig.java:27`

`GenericJackson2JsonRedisSerializer` 在 JSON 中嵌入 `@class` 类型信息，存在反序列化攻击风险。

**修复**：改用 `Jackson2JsonRedisSerializer` 并指定明确类型，或配置 `ObjectMapper` 的 `DefaultTyping` 白名单。

---

### 17. AuthService Turnstile 验证异常吞没

**位置**：`AuthService.java:38-51`

`catch (Exception e) {}` 吞没异常，Cloudflare 网络超时时注册直接放行（安全降级不当）。

**修复**：异常至少记录日志，并明确决定是放行还是拒绝。

## 低危问题（优化项）

### 18. 静态资源 setCachePeriod(0) 禁用浏览器缓存

**位置**：`WebConfig.java:16`

上传的图片每次请求都不走浏览器缓存，增加服务器带宽压力。

**修复**：设置 `setCachePeriod(86400)`（1 天）。

---

### 19. JWT 每次请求解析 3 次

**位置**：`JwtAuthFilter.java:28-33`

`validateToken` 解析一次，`getUserIdFromToken` 和 `getRoleFromToken` 各再解析一次 = 3 次 HMAC 验证。

**修复**：validate 后直接返回 `Claims` 对象，从中提取 userId 和 role。

---

### 20. ObjectMapper 每次方法调用内 new

**位置**：`ItemService.mergeInfoJson()`

`ObjectMapper` 是重量级线程安全对象，应该复用。

**修复**：提取为 `final` 类字段或使用 Spring 注入的全局实例。

---

### 21. 所有缓存统一 10min TTL

**位置**：`RedisConfig.java:29-30`

`visibleTypes` 变更频率极低可以更长；`hotItems` 可能需要更短以保证实时性。

**修复**：使用 `RedisCacheManager.builder().withInitialCacheConfigurations(...)` 为不同缓存设置不同 TTL。

---

### 22. reviews 表无外键约束

## 小服务器生存路线图

```
第一步（立竿见影）
  ├── 新增数据库索引（V2 迁移脚本）
  ├── 列表接口加分页
  └── Item 列表用 DTO 排除大字段

第二步（防服务挂掉）
  ├── 所有 RestClient 加超时（connect 5s / read 15s）
  ├── 修复 2 处 SQL 注入
  └── 文件上传降至 10MB

第三步（资源优化）
  ├── ReviewService 改 SQL 聚合
  ├── N+1 批量化（batchUpdateStatus 等）
  └── Redis 连接池 + HikariCP 显式配置

第四步（稳定性）
  ├── RabbitMQ 加 DLQ + prefetch + 重试
  ├── 外部 API 并行化 + 断路器
  └── 静态资源浏览器缓存
```

每一步都独立可交付，建议按顺序逐步实施。
