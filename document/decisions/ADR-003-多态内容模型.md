# ADR-003: 多态内容模型（items + info_json）

## 状态
已接受（Accepted）

## 日期
2025-02-20

## 上下文

DemoNet 是一个**多品类聚合平台**，需要支持 10 个品类：
- **game**、**movie**、**anime**、**boardgame**、**figure**、**book**、**music**、**digital**、**coffee**、**offline**

每个品类的属性差异极大：
- **游戏**：开发商、发行商、价格、配置要求、DLC
- **电影**：导演、演员、上映日期、片长
- **书籍**：作者、出版社、ISBN、页数、EPUB 链接
- **咖啡**：产地、烘焙度、风味描述、冲煮参数

### 需求
- **统一表结构**：所有品类的条目存在同一张 `items` 表
- **灵活扩展**：新增品类无需修改表结构
- **类型安全**：后端和前端需要知道每个品类的专属字段

### 可选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **单一表 + EAV**（当前方案） | 灵活、无需修改表结构 | 查询复杂、类型安全差 |
| **单一表 + JSON 列**（当前方案） | 灵活、查询相对简单 | 无法在 SQL 层面约束字段 |
| **Table-Per-Type（TPT）** | 类型安全、查询高效 | 新增品类需要新建表、JOIN 复杂 |
| **NoSQL（MongoDB）** | 天然支持多态 | 放弃关系型数据库的 ACID、事务 |

## 决策

**使用单一 `items` 表 + `info_json` JSON 列存储品类专属字段**：

### 表结构
```sql
CREATE TABLE items (
  id          BIGINT PRIMARY KEY AUTO_INCREMENT,
  type        VARCHAR(32) NOT NULL,      -- game/movie/anime/...
  slug        VARCHAR(128) NOT NULL UNIQUE,
  title       VARCHAR(256) NOT NULL,
  description TEXT,
  cover_url   VARCHAR(512),
  poster_url  VARCHAR(512),
  info_json   JSON,                       -- ← 品类专属数据
  status      TINYINT DEFAULT 0,          -- 0=待审核, 1=已上线
  hot_boost   INT DEFAULT 0,
  created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

### `info_json` 示例

**游戏（game）**：
```json
{
  "developer": "Game Science",
  "publisher": "Game Science",
  "price": "CNY 268.00",
  "genre": "动作, 冒险, 角色扮演",
  "release_date": "2024 年 8 月 19 日",
  "platform": "Windows",
  "features": ["单人", "Steam 成就", "完全支持控制器"]
}
```

**咖啡（coffee）**：
```json
{
  "origin": "Ethiopia",
  "region": "oromia",
  "variety": "Bourbon",
  "process": "Natural / Dry",
  "altitude": "2000",
  "cupper_points": 8.67,
  "total_cup_points": 89.0,
  "flavor_notes": "花果香,明亮"
}
```

### 后端解析
使用 **`tools.jackson`**（`ObjectMapper`）解析 `info_json`：
```java
// FetchConsumer.autoTag()
JsonNode infoJson = objectMapper.readTree(item.getInfoJson());
if (infoJson.has("developer")) {
  tags.add("开发商:" + infoJson.get("developer").asText());
}
```

### 前端处理
```javascript
// GameCard.vue
const hasDemo = computed(() => {
  try { return JSON.parse(props.item.infoJson || '{}').demo_available }
  catch { return false }
})
```

## 替代方案被拒绝的原因

### Table-Per-Type（TPT）
- ❌ 10 个品类需要 10 张表，新增品类需要 DDL
- ❌ 查询需要 UNION 或 JOIN，复杂度高
- ❌ 品类专属字段的增删改需要数据库迁移

### MongoDB
- ❌ 放弃 MySQL 的 ACID 事务（如订单、支付场景）
- ❌ 团队对 MySQL 更熟悉，学习成本高
- ❌ 现有基础设施（Flyway、MyBatis-Plus）无法复用

### 单一表 + 固定列
- ❌ 10 个品类的字段差异大，需要 100+ 列
- ❌ 大量 NULL 值，浪费存储空间
- ❌ 新增品类需要修改表结构

## 后果

### ✅ 正面影响
- **极致灵活**：新增品类无需修改表结构
- **快速迭代**：外部 API 字段变化时，无需数据库迁移
- **多态查询**：通过 `type` 字段过滤品类

### ⚠️ 负面影响
- **类型安全差**：`info_json` 字段在 SQL 层面无法约束
- **查询复杂**：需要在应用层解析 JSON，无法在 SQL 层面过滤
- **索引缺失**：无法对 `info_json` 内部字段建索引（除非使用 MySQL 8.0 JSON 索引）

### 📋 优化方向
1. **MyBatis-Plus JSON 处理器**：实现 `info_json` 字段的自动映射到 DTO
2. **MySQL JSON 索引**：对高频查询字段（如 `developer`、`author`）建 JSON 索引
3. **Schema 验证**：在应用层添加 JSON Schema 校验（如 `@Valid`）
