# ADR-007: 缓存策略 — Redis 多级缓存

## 状态
已接受（Accepted）

## 日期
2025-03-15

## 上下文

DemoNet 是一个内容聚合平台，面临以下性能挑战：

- **热点数据**：热门物品（hot_items）、精选推荐（featured）被频繁访问
- **读多写少**：物品详情、评论列表的读取频率远高于写入
- **实时性要求不同**：
  - 品类列表（visibleTypes）：一天更新一次即可
  - 热门物品（hotItems）：5 分钟缓存即可
  - 用户认证（JWT）：无需缓存，每次验证
- **外部 API 调用**：SteamGridDB 图片、IGDB 游戏封面等外部资源需要缓存

### 关键需求
- **读性能**：降低数据库负载，提升响应速度
- **灵活性**：不同数据需要不同的 TTL（生存时间）
- **可靠性**：缓存失效不影响核心功能（降级策略）
- **类型安全**：支持 Java 对象的序列化/反序列化

### 可选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **Redis + Spring Cache（当前方案）** | 成熟、灵活、TTL 可配置 | 需要维护 Redis 服务 |
| **Caffeine（本地缓存）** | 性能极高、无网络开销 | 分布式部署时数据不一致、内存占用大 |
| **Ehcache** | 本地 + 分布式支持 | 配置复杂、不如 Spring Cache 简洁 |
| **无缓存** | 简单 | 数据库压力大、响应慢 |

## 决策

**使用 Redis + Spring Cache 注解实现多级缓存策略**：

### 缓存架构

```
用户请求 → Spring Cache → Redis（命中）→ 返回
              ↓（未命中）
          数据库查询 → 写入 Redis → 返回
```

### 缓存配置（RedisConfig.java）

```java
@Configuration
@EnableCaching
public class RedisConfig implements CachingConfigurer {

    @Bean
    public CacheManager cacheManager(RedisConnectionFactory factory) {
        // 默认 TTL：10 分钟
        RedisCacheConfiguration config = RedisCacheConfiguration.defaultCacheConfig()
            .entryTtl(Duration.ofMinutes(10))
            .serializeValuesWith(RedisSerializationContext.SerializationPair
                .fromSerializer(new GenericJacksonJsonRedisSerializer(mapper)));

        // 不同缓存名称使用不同 TTL
        Map<String, RedisCacheConfiguration> initCaches = new HashMap<>();
        initCaches.put("visibleTypes", config.entryTtl(Duration.ofDays(1)));       // 品类列表：1 天
        initCaches.put("hotItems", config.entryTtl(Duration.ofMinutes(5)));        // 热门物品：5 分钟
        initCaches.put("featured", config.entryTtl(Duration.ofMinutes(5)));        // 精选推荐：5 分钟
        initCaches.put("recommended", config.entryTtl(Duration.ofMinutes(15)));    // 推荐列表：15 分钟

        return RedisCacheManager.builder(factory)
            .cacheDefaults(config)
            .withInitialCacheConfigurations(initCaches)
            .build();
    }
}
```

### 序列化策略

**使用 Jackson JSON 序列化**（而非 JDK 序列化）：

```java
// ✅ 推荐：Jackson JSON 序列化
GenericJacksonJsonRedisSerializer(mapper)

// ❌ 不推荐：JDK 序列化（体积大、不跨语言）
JdkSerializationRedisSerializer
```

**优势**：
- **体积小**：JSON 比 JDK 序列化小 50-70%
- **可读性**：Redis CLI 可直接查看缓存内容
- **跨语言**：其他语言（Python、Node.js）可以读取缓存
- **类型安全**：通过 `@Cacheable` 注解的泛型推断返回类型

### 使用示例

```java
@Service
public class ItemService {
    // 缓存热门物品，5 分钟 TTL
    @Cacheable(value = "hotItems", key = "'top_' + #limit")
    public List<Item> getHotItems(int limit) {
        return itemMapper.selectHotItems(limit);
    }

    // 缓存品类列表，1 天 TTL
    @Cacheable(value = "visibleTypes")
    public List<String> getVisibleTypes() {
        return itemMapper.selectDistinctTypes();
    }

    // 更新后清除缓存
    @CacheEvict(value = "hotItems", allEntries = true)
    public void refreshHotItems() {
        // 触发缓存刷新
    }
}
```

### 缓存错误处理

```java
@Override
public CacheErrorHandler errorHandler() {
    return new CacheErrorHandler() {
        @Override
        public void handleCacheGetError(RuntimeException e, Cache cache, Object key) {
            log.warn("Cache get error on {}: {}", cache.getName(), e.getMessage());
            // 降级策略：缓存失败不影响业务，返回空或查询数据库
        }

        @Override
        public void handleCachePutError(RuntimeException e, Cache cache, Object key, Object value) {
            log.warn("Cache put error on {}: {}", cache.getName(), e.getMessage());
            // 降级策略：写入失败不影响业务
        }
    };
}
```

## 替代方案被拒绝的原因

### Caffeine（本地缓存）
- ❌ **分布式部署时数据不一致**：多实例部署时，每个实例有独立的缓存
- ❌ **内存占用大**：缓存数据存在 JVM 堆内存，增加 GC 压力
- ❌ **无持久化**：服务重启后缓存清空

### Ehcache
- ❌ **配置复杂**：需要 XML 或 Java 配置，不如 Spring Cache 简洁
- ❌ **生态不如 Redis**：遇到问题解决方案少
- ❌ **分布式支持弱**：Ehcache 3.x 的分布式功能不如 Redis 成熟

### 无缓存
- ❌ **数据库压力大**：热门接口（如首页、热门列表）每秒请求几百次，数据库无法承受
- ❌ **响应慢**：数据库查询延迟 10-50ms，Redis 缓存延迟 <1ms

## 后果

### ✅ 正面影响
- **性能提升**：Redis 缓存延迟 <1ms，比数据库查询快 10-50 倍
- **灵活配置**：不同数据使用不同 TTL，平衡实时性和性能
- **降级策略**：缓存失败时不影响核心业务
- **可观测性**：通过 Redis CLI 或 Spring Boot Actuator 监控缓存命中率

### ⚠️ 负面影响
- **基础设施复杂度**：需要维护 Redis 服务（高可用、持久化、备份）
- **缓存一致性问题**：数据更新后需要主动清除缓存（或使用较短的 TTL）
- **内存占用**：Redis 缓存大量数据时，需要合理设置 `maxmemory` 和淘汰策略

### 📋 优化方向
1. **缓存监控**：通过 Spring Boot Actuator 的 `cache` 端点监控缓存命中率
2. **缓存预热**：应用启动时预加载热点数据到缓存
3. **分布式锁**：使用 Redis 分布式锁防止缓存击穿
4. **缓存穿透防护**：使用布隆过滤器（Bloom Filter）防止恶意请求穿透缓存
