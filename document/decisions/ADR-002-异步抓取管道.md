# ADR-002: 使用 RabbitMQ 异步抓取外部数据

## 状态
已接受（Accepted）

## 日期
2025-02-10

## 上下文

DemoNet 需要从 7 个外部 API 抓取数据：
- **Steam**（游戏）、**TMDB**（电影/动漫）、**IGDB**（游戏）、**AniList**（动漫）、**Bangumi**（动漫）、**iTunes**（音乐）、**SteamGridDB**（海报）

### 需求
- **异步处理**：抓取耗时（几秒到几十秒），不能阻塞用户请求
- **可靠性**：抓取失败需要重试，不能丢数据
- **解耦**：抓取逻辑与业务逻辑分离
- **可监控**：需要监控队列状态、失败率

### 可选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **同步抓取** | 简单、无基础设施 | 阻塞用户请求、可靠性差、无法解耦 |
| **RabbitMQ（当前方案）** | 成熟、可靠、支持重试/DLQ | 需要维护消息队列基础设施 |
| **Kafka** | 高吞吐、持久化 | 复杂度高、运维成本高 |
| **Redis Queue** | 轻量、无额外服务 | 可靠性不如 RabbitMQ、无原生重试机制 |
| **Quartz 定时任务** | 简单、内置调度 | 不适合异步任务、可靠性差 |

## 决策

**使用 RabbitMQ 作为异步消息队列，实现 7 个独立的抓取队列**：

### 架构设计
```
[AdminService] → [RabbitMQ: 7 queues] → [FetchConsumer] → [ItemService.saveItems]
                    ↓
              [DLX Exchange]
                    ↓
              [Dead Letter Queue]
```

### 队列配置
- **7 个队列**：`fetch.steam`、`fetch.tmdb.movie`、`fetch.tmdb.tv`、`fetch.igdb`、`fetch.anilist`、`fetch.bangumi`、`fetch.itunes`
- **预取计数**：`prefetch=5`（每消费者最多 5 条未确认消息）
- **重试机制**：最大 3 次（`max-attempts=3`）
- **死信队列**：3 次重试失败后进入 DLQ（`demonet.dlq`）

### 消息格式
```json
{
  "externalId": "730",  // Steam App ID 或 TMDB ID
  "type": "game"         // 品类
}
```

## 替代方案被拒绝的原因

### 同步抓取
- ❌ 管理员触发抓取后，HTTP 请求会阻塞 10-30 秒
- ❌ 网络超时会导致请求失败，用户体验差
- ❌ 无法实现重试和错误恢复

### Kafka
- ❌ DemoNet 的抓取量级（每天几百到几千条）不需要 Kafka 的高吞吐
- ❌ Kafka 运维复杂度高（需要 Zookeeper/Kraft、Topic 管理）
- ❌ 消息格式过于灵活，容易出错

### Redis Queue
- ❌ 无原生重试机制（需要手动实现）
- ❌ 无死信队列（需要手动处理失败消息）
- ❌ 可靠性不如 RabbitMQ（数据持久化能力弱）

## 后果

### ✅ 正面影响
- **异步解耦**：抓取逻辑不阻塞用户请求
- **可靠性**：DLQ 保证失败消息不丢失，便于后续人工处理
- **可监控**：RabbitMQ Management UI 可查看队列状态
- **易扩展**：新增抓取源只需新增队列和 Consumer

### ⚠️ 负面影响
- **基础设施复杂度**：需要维护 RabbitMQ 服务
- **调试困难**：异步流程追踪比同步复杂
- **最终一致性**：抓取后数据入库有延迟（几秒到几分钟）

### 📋 后续优化
1. **监控告警**：队列堆积超过阈值时触发告警
2. **幂等性**：`saveItems` 通过 `slug` 去重，避免重复数据
3. **批量抓取**：支持一次抓取多个 ID（当前仅支持单个 ID）
