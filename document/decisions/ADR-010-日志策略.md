# ADR-010: 日志策略 — SLF4J + 结构化日志

## 状态
已接受（Accepted）

## 日期
2025-04-01

## 上下文

DemoNet 需要日志系统满足以下需求：

- **可观测性**：需要追踪请求链路、异常堆栈、性能指标
- **生产调试**：线上出现问题时需要快速定位
- **审计合规**：管理员操作需要留痕（未来需求）
- **性能影响**：日志写入不能显著降低系统性能

### 关键需求
- **结构化日志**：易于检索和分析（如 ELK Stack）
- **日志级别**：ERROR（异常）、WARN（警告）、INFO（关键流程）、DEBUG（调试）
- **上下文追踪**：每个请求需要唯一的追踪 ID
- **性能优化**：异步日志、避免字符串拼接

### 可选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **SLF4J + Logback（当前方案）** | Spring Boot 默认、成熟、性能好 | 需要手动配置 |
| **Log4j2** | 性能比 Logback 更好 | 与 SLF4J 重复、Spring Boot 默认是 Logback |
| **System.out/err** | 简单 | 无结构化、无级别控制、性能差 |

## 决策

**使用 SLF4J + Logback 作为日志框架，Spring Boot 默认配置**：

### 1. 日志声明

```java
import lombok.extern.slf4j.Slf4j;

@Slf4j  // ✅ 推荐：Lombok 自动生成 log 字段
@RestController
public class ItemController {
    public List<Item> getHotItems(int limit) {
        log.info("Fetching hot items: limit={}", limit);  // 参数化日志
        try {
            // ...
        } catch (Exception e) {
            log.error("Failed to fetch hot items: {}", e.getMessage(), e);  // 包含异常堆栈
        }
    }
}
```

**禁止使用字符串拼接**：
```java
// ❌ 错误：字符串拼接即使日志级别不够也会执行
log.info("User " + userId + " fetched item " + itemId);

// ✅ 正确：参数化日志，只有在 INFO 级别启用时才执行拼接
log.info("User {} fetched item {}", userId, itemId);
```

### 2. 日志级别规范

| 级别 | 用途 | 示例 |
|------|------|------|
| **ERROR** | 系统异常、需要人工干预 | 数据库连接失败、第三方 API 调用失败 |
| **WARN** | 可恢复的错误、降级场景 | 缓存失效、重试机制触发 |
| **INFO** | 关键业务流程 | 用户登录、物品抓取、管理员操作 |
| **DEBUG** | 开发调试信息 | 请求参数、中间结果 |

**生产环境配置**：
```yaml
# application-prod.yml
logging:
  level:
    root: INFO
    com.example.demonet: INFO
    org.springframework.web: WARN
    org.hibernate.SQL: WARN  # 生产环境不输出 SQL 日志
```

**开发环境配置**：
```yaml
# application-dev.yml
logging:
  level:
    root: DEBUG
    org.springframework.web: DEBUG
    org.hibernate.SQL: DEBUG  # 开发环境输出 SQL 日志便于调试
```

### 3. 结构化日志

**使用占位符输出结构化数据**：
```java
// ✅ 结构化日志：每个占位符是一个独立的字段
log.info("User login: userId={}, ip={}, userAgent={}", userId, ip, userAgent);

// 输出示例（JSON 格式）：
// {"timestamp":"2025-04-01T10:30:00.123Z","level":"INFO","message":"User login","userId":123,"ip":"192.168.1.1","userAgent":"Mozilla/5.0"}
```

**MDC（Mapped Diagnostic Context）追踪请求链路**：
```java
@Component
public class RequestLoggingFilter extends OncePerRequestFilter {
    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain filterChain) throws ServletException, IOException {
        try {
            // 设置追踪 ID
            String traceId = UUID.randomUUID().toString();
            MDC.put("traceId", traceId);

            log.info("Request: method={}, uri={}", request.getMethod(), request.getRequestURI());
            filterChain.doFilter(request, response);
        } finally {
            MDC.clear();  // 清理 MDC，避免内存泄漏
        }
    }
}
```

### 4. 日志配置（logback-spring.xml）

```xml
<configuration>
    <springProfile name="prod">
        <!-- 生产环境：输出到文件 + 异步日志 -->
        <appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
            <file>logs/app.log</file>
            <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
                <fileNamePattern>logs/app.%d{yyyy-MM-dd}.log</fileNamePattern>
                <maxHistory>30</maxHistory>
            </rollingPolicy>
            <encoder>
                <pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] %-5level %logger{36} - %msg%n</pattern>
            </encoder>
        </appender>

        <appender name="ASYNC" class="ch.qos.logback.classic.AsyncAppender">
            <appender-ref ref="FILE" />
            <queueSize>512</queueSize>
            <discardingThreshold>0</discardingThreshold>
        </appender>

        <root level="INFO">
            <appender-ref ref="ASYNC" />
        </root>
    </springProfile>
</configuration>
```

### 5. 异常日志规范

**必须包含异常堆栈**：
```java
// ✅ 正确：第三个参数是 Throwable，会打印完整堆栈
log.error("Failed to fetch item: slug={}", slug, e);

// ❌ 错误：只打印异常消息，无堆栈
log.error("Failed to fetch item: " + e.getMessage());
```

**禁止吞掉异常**：
```java
// ❌ 错误：静默吞掉异常，无法追踪问题
try {
    // ...
} catch (Exception e) {
    // empty
}

// ✅ 正确：记录异常日志
try {
    // ...
} catch (Exception e) {
    log.warn("Non-critical operation failed: {}", e.getMessage());
}
```

## 替代方案被拒绝的原因

### Log4j2
- ❌ **与 SLF4J 重复**：SLF4J 已经是事实标准，Log4j2 的 API 与 SLF4J 重叠
- ❌ **Spring Boot 默认**：Spring Boot 默认使用 Logback，无需额外配置
- ❌ **迁移成本**：从 Logback 迁移到 Log4j2 需要修改配置和依赖

### System.out/err
- ❌ **无结构化**：无法按字段检索、分析
- ❌ **无日志级别**：无法区分 INFO/ERROR/WARN
- ❌ **性能差**：同步输出到控制台，阻塞线程

## 后果

### ✅ 正面影响
- **结构化日志**：易于检索、分析（可与 ELK Stack 集成）
- **性能优化**：参数化日志、异步日志、日志级别控制
- **可维护性**：Lombok `@Slf4j` 简化代码、统一规范
- **可观测性**：MDC 追踪请求链路，便于排查问题

### ⚠️ 负面影响
- **配置复杂**：生产/开发环境需要不同的日志配置
- **磁盘占用**：生产环境日志文件可能占用大量磁盘空间（需要日志轮转策略）
- **性能开销**：异步日志虽然性能好，但可能丢失日志（队列满时丢弃）

### 📋 优化方向
1. **日志聚合**：使用 ELK Stack（Elasticsearch + Logstash + Kibana）或 Loki + Grafana 集中管理日志
2. **审计日志**：管理员操作记录审计日志（表 `audit_logs`，记录 `who/when/what`）
3. **日志脱敏**：敏感数据（密码、Token）需要脱敏后再输出到日志
